Le contexte

Un repo. Toute ta startup.

Le problème

Tout part d'une contrainte unique.

La fenêtre de contexte se remplit vite. La qualité chute avec elle. Tout le reste en découle.

— Doctrine §1

Architecture

Trois axes, trois logiques.

Organiser par feature, par sujet et par output — jamais par rôle. Les rôles (relire, investiguer) deviennent des subagents dans une session, pas des sessions à part.

Découverte

Market · User · Support. Ce qui alimente le build ET l'audience.

  • /researchrecherche marché
  • /feedbackéchanges 1-à-1
  • /supportsift des tickets clients
→ knowledge/

Build

Spec → Code → Test. Par feature, en pipeline, avec gate humain au jalon. Les bugs trouvés en chemin deviennent des tickets dans bugs/, lus par /code comme des mini-specs.

  • /specle quoi + critères
  • /codele comment, étape par étape
  • /teste2e à œil neuf
→ features/ · bugs/

Audience

Post · Article · Newsletter · Report. Brand book partagé, drafts → review → posted. /report pull les stats via le MCP du réseau et synthétise ce qui marche pour informer les prochains drafts.

  • /postformat court (LinkedIn, X)
  • /articlelong form (blog)
  • /newsletterédition périodique
  • /reportanalytics via MCP réseau
→ content/
Comment ça marche · bout en bout

Trois sessions. Un fichier lu deux fois.

/spec écrit les critères. /code les lit pour construire. /test les lit pour valider à œil neuf. Les sessions passent ; les fichiers (et les critères) restent l'ancre commune.

session / fichier persistant lecture (reads) gate humain

Le secret du découpage : /test n'a pas écrit le code. Il lit le même SPEC que /code a lu — donc ses tests s'ancrent sur l'intention, pas sur le code écrit. C'est ce qui empêche les tests de ratifier les bugs.

Et les bugs trouvés en chemin ? Quand /test ou /support tombe sur un bug net, il dépose bugs/<slug>/TICKET.md — repro + critère « ne se reproduit plus ». La même session /code bugs/<slug> le lit comme une mini-SPEC et réutilise tout le pipeline (plan → étapes → hook → commit). Pas un 4e axe, juste un dossier d'artefacts.

Une journée type

Quatre moments. Quatre sessions. Des fichiers qui s'accumulent.

Le kit n'est pas un mode de travail à part — c'est juste la suite logique de ce que tu fais déjà. Chaque session laisse un artefact que la prochaine relit. Les sessions passent ; les fichiers restent.

Lundi 09:24 · Build

Tu reprends une feature en cours.

Tu lis ton SPEC.md de vendredi, et tu enchaînes sur la prochaine étape. Le hook lance les tests rapides à chaque commit. Tu ne réexpliques rien à Claude.

>/code checkout-flow
Mardi 14:10 · Discover

Tu sors d'un appel avec un user.

Tu notes ce qui ressort. Claude range dans knowledge/crm/contacts/, repère si un motif récurrent émerge, et propose de l'ajouter à insights.md.

>/feedback laurent
Mercredi 11:00 · Fix

Un bug est tombé hier soir.

Le ticket est déjà dans bugs/safari-cancel-crash/TICKET.md — déposé par /test à la fin du sprint. /code le lit comme une mini-SPEC, fixe, écrit le test de régression, ferme.

>/code bugs/safari-cancel-crash

Tu pars d'un insight users récent. Le brand book applique ta voix. Le draft va dans content/linkedin/drafts/ ; après ta review, il bascule en posted/.

Anti-perte

Hyper prudent par défaut.

/setup ne modifie jamais ton repo sans ta validation explicite. Quatre filets, sur tous les chemins.

Rollback en un commit

Un tag pre-setup-<timestamp> est posé avant toute action. Si quelque chose te gêne, un git reset ramène l'état d'origine.

Aucun fichier orphelin

Carte de migration old → new validée avant exécution. Les obsolètes vont dans _archive/, jamais en rm direct.

Code applicatif intact

Par défaut, git diff reste vide sur les chemins applicatifs. Seul le workflow autour évolue, sauf décision explicite de ta part.

Jamais de push auto

Tu décides toi-même quand tout est OK pour pousser. Les branches sensibles (main, production) demandent une double confirmation.

Install

Deux commandes. C'est tout.

Les skills sont chargés globalement dans ~/.claude/skills/ et deviennent disponibles dans toutes tes sessions Claude Code.

# 1. récupérer le kit $ gh repo clone lgrante/ai-founder-workflow ~/ai-founder-workflow # 2. installer les skills globalement $ ~/ai-founder-workflow/install.sh --global ✓ Skills installés : setup spec code test research feedback support post article newsletter report

Puis dans ton repo cible, ouvre Claude Code et tape /setup. Le skill pilote le déploiement, phase par phase, avec ton OK à chaque étape destructive.

Compagnons optionnels

nano-banana (skill global, wrap le Gemini CLI) — quand tu veux générer une image pour un post, /post & /article le détectent et proposent /generate. Coût ~0,04 $/image.
MCP par réseau (LinkedIn, Twitter/X, Substack…) — /report détecte le MCP du réseau via préfixe (mcp__linkedin__*, etc.) et pull les stats. Sans MCP, fallback export manuel.
Aucun n'est requis. Le kit fonctionne sans.
