ai-dev-workflow · kit Claude Code

Un workflow de dev
par feature, vérifié en continu

Des sessions Claude Code répétables, une vérification automatisée, un contexte qui reste propre. Déployable sur n'importe quel repo.

~/mon-repo $ /spec checkout-flow

le principe

Trois idées, une contrainte

Tout part d'un fait : la fenêtre de contexte se remplit vite, et la qualité baisse à mesure qu'elle se remplit. Le reste en découle.

01

La mémoire est dans les fichiers

Les sessions sont jetables. Le relais entre deux étapes est toujours un artefact durable : un spec, du code commité, des tests.

02

Une session = un contexte

On organise par feature (et par découverte), jamais par rôle. Les rôles deviennent des subagents dans une session.

03

La vérification est automatisée

Un hook lance les tests et bloque tant que c'est rouge. Tu n'es jamais le messager entre deux étapes.

À retenir : dès que ce qui compte est dans un fichier, une session neuve (ou /clear) bat une longue session encombrée.

organisation

Deux axes de sessions

D

Découverte

market-research- · user-feedback-

  • Deux types distincts : marché (extérieur) et utilisateurs réels (intérieur).
  • Continu, jamais rattaché à une feature.
  • Sort dans des fichiers de connaissance. C'est la source des specs.
B

Build

spec- → code- → test-

  • Par feature, en pipeline en trois temps.
  • Chaque temps produit un artefact que le suivant lit.
la découverte alimente les specs du build

les skills

Les commandes, et comment elles marchent

Chaque type de session a une commande (un skill dans .claude/skills/<nom>/SKILL.md, invocable en /<nom>). Elle charge le bon contexte et te met sur les bons rails. Voici ce qui se passe quand tu en tapes une :

anatomie d'une commande — exemple : /code
01
Tu tapes /code checkout-flowdans ta session Claude Code, à la racine du repo.
02
Claude Code lit .claude/skills/code/SKILL.md et remplace $ARGUMENTS par checkout-flow.Le contenu du skill devient le prompt de Claude.
03
Le skill te rappelle de nommer la session : /rename code-checkout-flowClaude ne peut pas lancer /rename lui-même — c'est ton geste.
04
Le skill charge le contexte et pilote la suite : lit features/checkout-flow/SPEC.md, planifie, puis implémente étape par étape.Le Stop hook teste à chaque étape.

Les cinq commandes, deux par axe pour la découverte, trois pour le build :

/researchmarket-research-

Recherche marché, concurrents, tendances. Continue, jamais rattachée à une feature. Délègue à un subagent pour garder le contexte propre.

→ knowledge/market/ + insights.md

/feedbackuser-feedback-

Échange avec un utilisateur réel. Charge sa fiche, puis met à jour la fiche CRM et l'agrégat d'insights. Les idées émergent de l'agrégat.

→ knowledge/crm/contacts/ + insights.md

/specspec-

Écrit le SPEC : description + critères d'acceptation (la définition de « fini », avant le code) + jalons. Ne code pas.

→ features/<feature>/SPEC.md

/codecode-

Planifie (PLAN.md) puis implémente étape par étape. Un subagent écrit le filet rapide depuis les critères ; le hook le lance et bloque tant que c'est rouge.

→ PLAN.md + code commité

/testtest-

Session fraîche qui n'a pas écrit le code. Écrit la suite e2e depuis le spec, au jalon, et fait la revue à œil neuf. Sa référence est le SPEC, pas le code.

→ suite e2e commitée

Le format des préfixes (market-research-…, MARKET_RESEARCH_…) est libre ; seul le préfixe par type compte, pour retrouver les sessions avec --resume.

le pipeline

Le cycle d'une feature

spec définit le quoi → code construit avec son filet rapide → test valide indépendamment → tu signes le jalon.

Une feature = une branche git. Les trois sessions build (spec-, code-, test-) travaillent toutes sur la même branche dédiée (ex. feature/<feature>) — jamais deux features mélangées. Plusieurs features en parallèle → worktrees git séparés. La découverte n'est rattachée à aucune branche.

vérification

Deux sortes de tests, deux boulots

La ligne de partage n'est pas « unitaire vs le reste ». C'est rapide & incrémental (à l'étape) vs bout-en-bout (au jalon).

①

Filet rapide

chez code-x · à chaque étape

  • Unitaires + intégration rapide.
  • Écrits par un subagent, depuis les critères de l'étape.
  • Lancés par le hook, qui bloque tant que c'est rouge → code-x corrige seule.
  • Filet de régression, pas validation finale.

Le cuisinier qui goûte son plat en cuisinant.

②

Validation indépendante

chez test-x · au jalon

  • e2e + acceptation.
  • Écrits depuis le spec, par une session qui n'a pas écrit le code.
  • Ne se laisse pas avoir par ses propres suppositions.
  • Inclut la revue à œil neuf.

Le critique qui juge l'assiette finie face au menu.

tests ancrés sur l'intention · pas 100 % de couverture · disjoncteur après 2-3 échecs

le hook

Comment test-gate.sh tient la boucle

C'est un Stop hook : il se déclenche quand code-x finit un tour. Sa logique tient en quatre temps.

.claude/hooks/test-gate.sh
01
Lit stop_hook_active. S'il est vrai (relance déjà déclenchée par ce hook), il laisse finir.Garde anti-boucle obligatoire — sinon un test rouge bloquerait à l'infini.
02
Lance TEST_CMD (ta commande de tests rapides) et écrit la sortie dans .cc-scratch/.
03
Vert → exit 0 : code-x peut terminer l'étape.
04
Rouge → renvoie {"decision":"block","reason":…} : Claude ne peut pas finir, lit le fichier de résultats et corrige.
Un seul réglage à faire par repo : la variable TEST_CMD en haut du script (et la même commande dans CLAUDE.md).

structure

Deux arborescences à ne pas confondre

Le repo kit (ce qu'on partage)

ai-dev-workflow/
├── README.md · README.html     # le guide
├── DEPLOY.md                  # brief à donner à une session CC (repo cible)
├── CONTRIBUTING.md · LICENSE   # contribuer (PR, doctrine) · MIT
├── templates/                 # ce qui s'installe
│   ├── CLAUDE.md  ·  docs/WORKFLOW.md
│   └── .claude/{settings.json, hooks/test-gate.sh, skills/{spec,code,test,research,feedback}}
│          # + opt-in non activés : hooks/context-{handoff,restore}.sh, statusline.sh
├── scaffold/                  # arbo vide : knowledge/, features/, .cc-scratch/
├── examples/checkout-flow/     # exemple travaillé : SPEC.md + PLAN.md
└── install.sh

Ce que le kit déploie dans ton repo

<ton repo>/
├── CLAUDE.md                  # court : commandes, conventions
├── .claude/                   # hook + commandes /spec /code /test /research /feedback
├── docs/WORKFLOW.md
├── knowledge/                 # DÉCOUVERTE : market/, insights.md, crm/contacts/
├── features/<feature>/      # BUILD : SPEC.md + PLAN.md
├── .cc-scratch/               # gitignored — résultats de tests transitoires
└── <code applicatif>          # INCHANGÉ

déployer

Deux voies, selon le repo

$

Repo quasi vierge

install.sh

  • Copie templates + scaffold sans rien écraser, ajoute au .gitignore.
  • Tu renseignes TEST_CMD et le CLAUDE.md.
▸

Repo existant

DEPLOY.md → session Claude Code

  • Tu donnes DEPLOY.md à une session ; elle inventorie, propose un mapping + carte de migration, et ne touche rien avant ta validation.
  • Migration tracée (git mv, _archive/), aucune perte.
Non-négociable : mémoire dans les fichiers, organisation par feature + découverte, pipeline spec→code→test, gate humain au jalon. Adaptable : noms de dossiers, stack, frameworks de test, format des préfixes.
